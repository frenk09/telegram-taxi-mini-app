# Архитектура MVP

## 1. Контур системы

Сервис состоит из четырёх основных блоков:

- Telegram Bot API layer;
- backend API с доменной логикой заказов;
- mini app для клиента и водителя;
- админский контур для модерации и операционного контроля.

## 2. Предлагаемый runtime-поток

1. Пользователь открывает бота.
2. Бот определяет роль и открывает mini app либо соответствующий workflow.
3. Mini app отправляет данные заказа в backend API.
4. Backend рассчитывает цену и создаёт заказ.
5. Заказ рассылается активным водителям в том же городе.
6. Первый принявший водитель атомарно закрепляется за заказом.
7. По ходу поездки водитель переводит заказ по статусам.
8. Backend отправляет уведомления клиенту, водителю и при необходимости администратору.

## 3. MVP-упрощения

- распределение заказов по городу, без live-геопоиска ближайшего водителя;
- хранение координат и адресов уже в нормализованном виде;
- простая тарифная модель без surge pricing;
- административное ручное вмешательство доступно на любом этапе.

## 4. Следующий этап развития

- Webhook/polling слой для Telegram;
- PostgreSQL repositories и Redis-кэш;
- файлы документов водителя через S3-совместимое хранилище;
- websocket/SSE для real-time статусов;
- antifraud и рейтинг пользователей;
- платёжный шлюз и удержание комиссии.
